# review-pr

## 触发条件
- 用户要求审查 PR
- 用户说 "review" 或 "审查" 或 "看看这个 PR"
- 用户提供 PR 链接或编号

## 指令
1. 获取 PR 的基本信息（标题、描述、作者）
2. 获取 PR 的文件变更列表
3. 逐个审查关键文件的改动
4. 检查以下方面：
   - 代码逻辑是否正确
   - 是否有潜在的 bug
   - 是否有安全风险
   - 是否符合项目代码规范
   - 测试覆盖是否充分
5. 生成结构化的审查报告

## 示例
用户：帮我审查 PR #42
助手：我来审查这个 PR...

## PR #42: Add user authentication

### 变更文件
- src/auth/login.py (+120/-5)
- src/auth/middleware.py (+45/-0)
- tests/test_auth.py (+80/-0)

### 审查意见

**问题 (2)**
1. `login.py:45` - 密码明文比较，应使用 bcrypt
2. `middleware.py:23` - 缺少 token 过期检查

**建议 (1)**
1. 考虑添加登录失败次数限制

**优点**
- 测试覆盖良好
- 代码结构清晰

总体评价：需要修改后合并
